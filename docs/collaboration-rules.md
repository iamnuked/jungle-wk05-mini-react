# WEEK5 협업 룰 - Mini React Dog Breed Quiz

이 문서는 [`requirements.md`](./requirements.md), [`architecture.md`](./architecture.md), [`api-spec.md`](./api-spec.md)를 기준으로 4인 협업 시 충돌을 줄이고 구현 속도를 높이기 위한 협업 규칙입니다.

## 1. 협업 원칙

- 모든 구현은 문서 기준으로 진행한다.
- 문서에 없는 설계 변경은 바로 코드로 들어가지 않고 먼저 팀 합의를 거친다.
- 코어 엔진과 앱 로직은 섞지 않는다.
- 각자 맡은 책임 범위를 넘는 수정은 최소화한다.
- 다른 사람의 작업 파일을 수정해야 할 경우 먼저 공유한다.
- 작게 자주 머지하고, 큰 덩어리 브랜치는 오래 끌지 않는다.

## 2. 폴더 책임 규칙

폴더별 책임은 아래 기준으로 고정합니다.

- `src/mini-react/`
  - Mini React 코어
  - `FunctionComponent`, hooks, renderer, event, vdom, diff, patch
- `src/components/`
  - UI 컴포넌트
  - 상태 없는 순수 함수형 컴포넌트만 작성
- `src/services/`
  - 외부 API 호출
  - Dog CEO API 요청과 응답 검증
- `src/domain/`
  - 퀴즈 규칙
  - 문제 생성
  - 정답 판정
  - 입력 정규화
- `tests/`
  - 각 레이어 테스트

이 규칙을 깨면 코드 책임이 흐려지고 충돌이 커지므로, 파일 추가도 이 기준을 따릅니다.

## 3. 상태 관리 규칙

- 상태는 `App` 루트 컴포넌트에서만 관리한다.
- 자식 컴포넌트는 local state를 만들지 않는다.
- 자식 컴포넌트는 `props`만 받아 화면만 그린다.
- API 호출 결과를 자식 컴포넌트 안에서 직접 처리하지 않는다.

## 4. 컴포넌트 작성 규칙

- 모든 컴포넌트는 함수형 컴포넌트로 구현한다.
- JSX는 사용하지 않는다.
- 컴포넌트는 순수 `vnode 객체`를 반환한다.
- 컴포넌트에서 `fetch()`를 직접 호출하지 않는다.
- 컴포넌트에서 정답 판정 로직을 직접 구현하지 않는다.
- 컴포넌트는 화면 표현과 이벤트 전달만 담당한다.

## 5. Mini React 코어 수정 규칙

- `FunctionComponent` 구조는 팀 합의 없이 바꾸지 않는다.
- `hooks[]`, `hookIndex`, `currentComponent` 계약은 고정한다.
- `mount()`와 `update()` 시그니처가 바뀌면 바로 공유한다.
- `useState`, `useEffect`, `useMemo`의 저장 구조를 바꿀 때는 관련 테스트를 같이 수정한다.
- diff/patch 엔진은 기존 WEEK4 동작을 최대한 유지한다.

## 6. 이벤트 처리 규칙

- 화면 상호작용은 모두 함수형 이벤트 props로 처리한다.
- inline event string은 앱 코드에서 사용하지 않는다.
- 이벤트 바인딩 규칙은 `src/mini-react/event.js`에만 둔다.
- 이벤트 핸들러 비교/교체 정책은 한 곳에서만 관리한다.

## 7. API 사용 규칙

- 외부 API는 `src/services/api.js`에서만 호출한다.
- 화면 컴포넌트나 `App`에서 직접 `fetch()`를 쓰지 않는다.
- 응답 검증 없이 raw JSON을 바로 state에 넣지 않는다.
- breed 데이터는 반드시 `BreedEntry` 형태로 변환한 뒤 사용한다.

## 8. 도메인 로직 규칙

- 문제 생성은 `src/domain/quiz.js`에서 처리한다.
- 정답 판정은 `src/domain/normalize.js`와 `src/domain/quiz.js`에서 처리한다.
- 이미지 URL 파싱으로 정답을 판정하지 않는다.
- 사용자 입력 비교는 반드시 정규화 후 처리한다.

## 9. 브랜치 규칙

- 브랜치는 역할별로 분리한다.
- 브랜치 이름은 아래 형식을 권장한다.

```text
member-a/mini-react-runtime
member-b/vdom-events
member-c/app-flow-domain
member-d/ui-components
```

- `main`에 직접 작업하지 않는다.
- 작업 시작 전 최신 `main` 기준으로 브랜치를 만든다.
- 머지 전에는 관련 테스트를 먼저 통과시킨다.

## 10. 머지 순서 규칙

이번 프로젝트는 아래 순서로 머지하는 것이 가장 안전합니다.

1. A: Mini React Runtime
2. B: VDOM / Diff / Patch / Event 연결
3. C: App 상태 흐름 + Domain + Service
4. D: UI 컴포넌트
5. 최종 통합 브랜치에서 테스트와 README 정리

이유:

- A, B가 없으면 앱이 렌더되지 않는다.
- C는 A, B의 인터페이스가 있어야 안정적으로 붙일 수 있다.
- D는 목업 데이터로 먼저 만들 수 있지만, 최종 연결은 C 인터페이스를 따라야 한다.

## 11. 인터페이스 합의 필수 항목

아래 항목은 구현 전에 반드시 팀이 먼저 합의합니다.

- `FunctionComponent` 생성 방식
- `mount()`와 `update()` 호출 방식
- vnode shape
- 이벤트 prop 표현 방식
- `BreedEntry` shape
- `currentQuestion` shape
- `App` state shape
- 정답 판정 함수 이름과 입력값

이 항목은 중간에 자주 바뀌면 충돌이 크게 납니다.

## 12. 코드 리뷰 규칙

- 리뷰는 스타일보다 동작과 계약을 먼저 본다.
- 아래 항목을 우선 체크한다.

1. 문서의 계약을 어겼는지
2. 다른 담당자 파일까지 불필요하게 건드렸는지
3. 책임 분리가 무너졌는지
4. 테스트가 빠졌는지
5. 상태 흐름이 루트 중심 구조를 깨는지

## 13. 테스트 규칙

- 기능 추가 시 최소 1개 이상 테스트를 함께 추가한다.
- 코어 수정 시 코어 테스트를 먼저 통과시킨다.
- 서비스 계층 수정 시 정상 응답과 실패 응답 둘 다 테스트한다.
- 도메인 로직 수정 시 정답 판정 케이스를 추가한다.
- UI 수정 시 최소한 렌더 결과 또는 이벤트 연결 테스트를 확인한다.

## 14. 충돌 방지 규칙

- 같은 파일을 두 명 이상이 동시에 수정하지 않도록 한다.
- 공용 인터페이스 파일은 담당자 1명이 우선 소유한다.
- 구조 변경은 먼저 문서에 반영하고 공유한 뒤 진행한다.
- `src/app.js`는 앱 흐름 담당자가 우선 소유한다.
- `src/mini-react/*`는 코어 담당자가 우선 소유한다.

## 15. 금지 사항

- 문서와 다른 구조를 임의로 추가하는 것
- 컴포넌트 내부에서 직접 API 호출하는 것
- 자식 컴포넌트에 state를 넣는 것
- 정답 판정을 UI 컴포넌트 안에서 처리하는 것
- 이벤트 로직을 여러 파일에 분산시키는 것
- 테스트 없이 코어 계약을 바꾸는 것

## 16. 최종 목표

협업의 목표는 단순히 파일을 나눠 갖는 것이 아니라, 각 레이어가 문서 기준으로 정확히 연결되도록 만드는 것입니다.

따라서 모든 팀원은 아래를 공통 기준으로 삼습니다.

- Mini React 코어는 `src/mini-react/`
- API는 `src/services/`
- 도메인 로직은 `src/domain/`
- 화면은 `src/components/`
- 상태는 `App`
