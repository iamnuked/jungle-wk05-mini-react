## WEEK5 요구사항 - Mini React Dog Breed Quiz

### 1. 공통 목표

- WEEK4에서 구현한 `Virtual DOM + Diff + Patch` 엔진을 재사용해야 한다.
- 그 위에 `Mini React runtime`을 추가해 실제로 동작하는 웹 앱을 완성해야 한다.
- 최종 결과물은 `Dog Breed Quiz` 앱이어야 한다.
- 제출물에는 실행 가능한 앱, 테스트 코드, README가 포함되어야 한다.

### 2. Mini React 구현 요구사항

- 함수형 컴포넌트를 감싸는 `FunctionComponent` 클래스를 직접 구현해야 한다.
- 각 `FunctionComponent` 인스턴스는 컴포넌트별 Hook 상태를 저장하는 `hooks` 배열을 가져야 한다.
- `FunctionComponent`는 최초 렌더링을 수행하는 `mount()` 메서드를 가져야 한다.
- `FunctionComponent`는 상태 변경 이후 다시 렌더링하는 `update()` 메서드를 가져야 한다.
- `update()`는 컴포넌트 함수를 다시 실행해 새로운 vnode를 만들고, 이전 vnode와 비교한 뒤 변경된 부분만 실제 DOM에 반영해야 한다.
- 모든 UI는 함수형 컴포넌트로 구현해야 한다.
- JSX는 사용하지 않고 순수 `vnode 객체`를 반환해야 한다.
- 함수형 컴포넌트는 diff 대상에 직접 넣지 않고, 렌더 단계에서 일반 `element/text vnode`로 해석되어야 한다.
- `useState`, `useEffect`, `useMemo`를 최소 기능으로 직접 구현해야 한다.
- Hook은 `hookIndex` 기반으로 동작해야 한다.
- Hook은 현재 실행 중인 `FunctionComponent` 인스턴스를 기준으로 동작해야 한다.
- Hook 값은 렌더가 다시 일어나도 유지되어야 한다.
- `useEffect`는 의존성 배열 비교와 cleanup을 지원해야 한다.
- `useMemo`는 의존성이 바뀌지 않으면 이전 계산 결과를 재사용해야 한다.

### 3. 상태 관리 요구사항

- 상태는 `App` 루트 컴포넌트에서만 관리해야 한다.
- 자식 컴포넌트는 local state를 가지지 않아야 한다.
- 자식 컴포넌트는 모두 `props`만 받는 순수 함수여야 한다.
- 상태 변경 시 루트 컴포넌트가 다시 실행되어 새로운 vnode 트리를 생성해야 한다.
- 새 vnode는 이전 vnode와 비교되어 변경된 부분만 실제 DOM에 반영되어야 한다.

### 4. 이벤트 처리 요구사항

- `onClick`, `onChange`, `onInput` 같은 함수형 이벤트 props를 지원해야 한다.
- 함수형 이벤트는 실제 DOM에 `addEventListener` 방식으로 연결되어야 한다.
- rerender 시 이벤트 핸들러가 변경되면 이전 리스너를 제거하고 새 리스너를 연결해야 한다.
- 앱의 사용자 상호작용은 inline event string이 아니라 함수형 이벤트 props 기반으로 구현해야 한다.

### 5. Dog Breed Quiz 앱 요구사항

- 시작 화면, 퀴즈 화면, 결과 화면이 있어야 한다.
- 시작 화면에서 문제 수를 선택할 수 있어야 한다.
- 시작 버튼을 누르면 퀴즈가 시작되어야 한다.
- 문제마다 강아지 이미지 1장과 정답을 입력할 수 있는 입력칸이 표시되어야 한다.
- 사용자는 입력칸에 견종명을 직접 입력할 수 있어야 한다.
- 제출 버튼을 누르면 입력값 기준으로 정답 여부를 판정해야 한다.
- 제출 직후 정답 또는 오답 여부가 화면에 표시되어야 한다.
- 피드백 이후 다음 버튼으로 다음 문제로 넘어갈 수 있어야 한다.
- 마지막 문제까지 끝나면 결과 화면으로 이동해야 한다.
- 결과 화면에는 점수와 정답률이 표시되어야 한다.
- 다시 하기 버튼으로 초기 상태로 돌아갈 수 있어야 한다.

### 6. API 및 데이터 처리 요구사항

- `Dog CEO API`를 사용해야 한다.
- 앱 시작 시 전체 견종 목록을 불러와야 한다.
- 견종 목록은 앱에서 사용하기 쉬운 평탄화 구조로 변환되어야 한다.
- 문제 생성은 `정답 후보를 먼저 선택한 뒤`, 그 정답에 맞는 이미지를 요청하는 방식으로 처리해야 한다.
- 정답 판정은 이미지 URL 파싱에만 의존하면 안 된다.
- 서브브리드가 있는 경우도 처리 가능해야 한다.
- 사용자 입력값은 정규화 후 정답과 비교할 수 있어야 한다.

### 7. 비동기 처리 요구사항

- `useEffect`를 이용해 초기 breed 목록 로드와 문제 로드를 처리해야 한다.
- 이전 문제의 늦은 API 응답이 현재 문제를 덮어쓰지 않도록 방지해야 한다.
- fetch 후처리는 cleanup 또는 guard로 안전하게 관리해야 한다.
- 사용자가 제출 버튼을 여러 번 눌러도 판정은 한 번만 이루어져야 한다.

### 8. 기존 엔진 연동 요구사항

- WEEK4의 VDOM 생성, diff, patch 코드를 최대한 재사용해야 한다.
- 기존 엔진은 WEEK5 구조에 맞게 필요한 범위에서만 재배치 또는 래핑해야 한다.
- Mini React 코어 코드는 `src/mini-react/` 아래에 구성해야 한다.
- 외부 API 호출 코드는 `src/services/`에 분리해야 한다.
- 퀴즈 규칙, 정답 판정, 입력 정규화 같은 도메인 로직은 `src/domain/`에 분리해야 한다.
- state 변경 시 전체 DOM을 통째로 다시 그리는 것이 아니라, 기존 diff/patch를 통해 변경 부분만 업데이트해야 한다.
- keyed diff 동작은 유지되어야 한다.

### 9. 테스트 요구사항

- Hook 테스트가 있어야 한다.
- `useState` 상태 유지 동작을 검증해야 한다.
- `useEffect` deps 비교와 cleanup 동작을 검증해야 한다.
- `useMemo` 캐싱 동작을 검증해야 한다.
- VDOM diff/patch 테스트가 있어야 한다.
- 텍스트 변경, props 변경, 자식 삽입/삭제, keyed 이동을 검증해야 한다.
- 이벤트 테스트가 있어야 한다.
- 함수형 이벤트 핸들러 연결 및 교체를 검증해야 한다.
- Quiz 유틸 테스트가 있어야 한다.
- breed 목록 평탄화, 입력 정규화, 문제 생성, 정답 판정을 검증해야 한다.
- 기능 테스트 수준에서 시작, 문제 진행, 입력 제출, 정답/오답 표시, 결과 화면, 다시 하기를 확인할 수 있어야 한다.
